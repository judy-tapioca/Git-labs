# Отчет по лабораторной работе №2 Git Flow, Code Review


### Подготовка репозитория

.gitignore — это файл, который указывает Git, какие файлы и папки следует игнорировать при отслеживании изменений. Любой файл или шаблон, указанный в нём, будет невидим для Git — он не появится в выводе git status и не будет случайно закоммичен. Чаще всего его используют для исключения файлов сборки (build/,*.exe)

### Защита веток
Branch protection rules — это настройки GitHub, которые ограничивают прямые изменения в защищённой ветке. С их помощью можно запретить прямой пуш в ветку, потребовать обязательного code review через Pull Request, а также заблокировать force push и удаление ветки. Это гарантирует, что в основную ветку попадает только проверенный код.
Настройка: Settings , Branches , Add branch protection rule , указать main , включить нужные ограничения.
После включения защиты git push origin main возвращает ошибку — изменения можно внести только через Pull Request из другой ветки.



Три основные команды:

1. `git add .`
2. `git commit -m "message"`
3. `git push origin main`

`git add .` — добавляет все изменённые файлы в staging area (подготовка к коммиту). `git commit -m "message"` — сохраняет эти файлы в локальную историю проекта с описанием изменений. `git push origin main` — отправляет сохранённые изменения в удалённый репозиторий на GitHub.

### Code Review

Pull Request (PR) — это запрос на слияние изменений из одной ветки в другую. PR позволяет просмотреть все изменения перед слиянием и провести code review. После одобрения PR ветки сливаются, а локальный репозиторий обновляется командой `git pull`.
